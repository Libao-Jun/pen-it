# 工具函数全流程

本 skill 用于为 `pen-it` 项目新增一个工具函数模块，覆盖从代码编写到文档更新的完整流程。

## 触发条件

当用户请求新增工具函数、添加 util 方法、或在 `src/` 下新建模块时触发。

## 项目约定速查

| 规则     | 说明                                                                         |
| -------- | ---------------------------------------------------------------------------- |
| 模块目录 | `src/<module>/index.ts`                                                      |
| 测试目录 | `tests/<module>.test.ts`                                                     |
| 入口导出 | `src/index.ts` 中添加 `export * from './<module>'`                           |
| 测试框架 | vitest，使用 `describe` / `it` / `expect`                                    |
| 类型检查 | `npx tsc --noEmit`                                                           |
| 运行测试 | `npx vitest run`（全量）或 `npx vitest run tests/<module>.test.ts`（单模块） |
| 注释风格 | JSDoc，中文简述，含 `@param` / `@returns` / `@example`，精简不冗余           |

## 参考现有代码

- 源码风格参考 → `src/array/index.ts`
- 测试风格参考 → `tests/array.test.ts`
- README API 文档格式参考 → `README.md` 中 `### 数组 — \`array\`` 段落
- 更新日志文档放入 CHANGELOG.md  中，后续更新日志参考现有的文档格式

## 流程步骤

按顺序执行以下 5 步，每步完成后确认无误再进入下一步。

### 第 1 步：生成工具函数

**输入**：用户描述的需求或函数名。

**操作**：

1. 确定函数所属模块（复用已有模块或新建模块）
2. 在 `src/<module>/index.ts` 中编写实现
3. 函数要求：
   - **纯函数**优先——不修改入参，不依赖外部状态
   - 类型安全——合理的 TypeScript 泛型约束
   - 精简——功能内聚，不过度抽象

**自查清单**：

- 是否遵循单向数据流（输入 → 输出）？
- 是否避免了不必要的副作用？
- TypeScript 类型是否推导正确？
- 边界情况是否处理（空值、极端值、异常输入）？

### 第 2 步：添加标准注释

**规则**：

1. 模块顶部用 `//` 写中文分类标题
2. 每个导出函数上方写 JSDoc：
   - 第一行：`/**` 后跟一句中文功能描述
   - `@param` — 参数说明（含默认值）
   - `@returns` — 返回值说明
   - `@example` — 一个精简用例（最多一个，控制台输出用 `// =>` 标注）
3. **禁止**：多段落叙述、场景罗列、实现原理注释、行内注释、类比比喻、`@example` 内写 fetch/addEventListener 等完整场景代码
4. 参考 `src/array/index.ts` 中的注释密度

### 第 3 步：生成测试用例

**在 `tests/<module>.test.ts` 中生成测试**：

1. 引入 vitest：`import { describe, it, expect, vi, beforeEach, afterEach } from 'vitest'`
2. 引入被测函数：`import { fn1, fn2 } from '../src/<module>'`
3. 用 `// ====` 分隔各函数的测试区块
4. 每个函数覆盖：
   - 基本功能（happy path）
   - 边界情况（空值、单元素、极端值）
   - 不修改原始数据（若为纯函数）
   - 各参数组合 / 分支路径
5. 若涉及定时器，使用 `vi.useFakeTimers()` + `vi.advanceTimersByTime()`

### 第 4 步：跑测试并修 bug

**操作**：

1. 先跑类型检查：`npx tsc --noEmit`
2. 再跑测试：`npx vitest run tests/<module>.test.ts`
3. 若失败，根据报错定位源码或测试的 bug
4. 修复后回到步骤 2（检查注释是否需要同步调整）
5. 确保全部通过后再跑全量：`npx vitest run`

**常见 bug 排查**：

- 测试期望写错（时序、调用次数）
- `lastArgs` 在闭包中未及时更新
- 边界条件（delay=0、空参数、连续快速调用）
- 状态残留（`beforeEach` 未正确重置）

### 第 5 步：更新 README API 文档

**在 `README.md` 中添加新模块的 API 段落**：

格式模板（严格遵循现有格式）：

````markdown
### 模块中文名 — `module`

- **`fnName(param)`** — 一句话功能描述。

```ts
fnName(arg);
// => result
```
````

**规则**：

- 用 `###` 标题，格式：`### 中文名 — \`module\``
- 每个函数一行：`- **\`fnName(param)\`\*\* — 描述。`
- 紧跟一个 ` ```ts ` 代码块展示用法和输出
- 若模块函数较多，可按子分类拆分为多个 `###` 标题（参考 `format` 模块）
- 新模块的段落插入到 `---` 分隔线之前，与已有模块保持一个空行间距

### 第 6 步：更新 CHANGELOG.md 文档

- 文档格式和规范参照现有的 `CHANGELOG.md` 更新日志文档规范

## 完成标志

全部 6 步执行完毕，`npx tsc --noEmit` 零错误，`npx vitest run` 全量通过，README 已更新，CHANGELOG 已更新。
